# Error Registry Schema and Snapshots

This directory contains the **canonical JSON Schema** for Seqlok's error
registry export, plus a set of **generated snapshots** derived from the
TypeScript error registries.

> ⚠️ **Do not hand-edit generated files.**
>
> Only `error-registry.schema.json` is meant to be edited by humans.
> All `error-registry.*.json` snapshot files are generated and can be
> safely overwritten by the CLI.

It is the single place non-TypeScript tooling should look when they want to
understand or consume the error universe.

---

## Files

### `error-registry.schema.json` (source-of-truth)

**Human-maintained.**

Defines the shape of a registry export produced by `@seqlok/introspect`.

Top-level fields:

- `schemaVersion` – numeric schema version (currently `1`)
- `generatedAt` – ISO-8601 timestamp for the export
- `generator` – identifier of the generator (`"@seqlok/introspect"`)
- `domainIds` – map of `domainPrefix -> numericDomainId`
- `domains` – per-domain view of codes
- `allCodes` – flattened list of all codes across all domains
- `stats` – summary counts by domain and severity

Every JSON snapshot generated by `scripts/export-error-registry.ts` is expected
to validate against this schema.

---

## Generated snapshots

These files are **generated artifacts**. Do not edit them manually; regenerate
them using the commands below.

### `error-registry.full.json`

Snapshot of **all registered errors** in the JS/TS universe:

- Includes every domain (`internal`, `primitives`, `env`, `spec`, `plan`,
  `backing`, `binding`, `handoff`, `introspect`, …)
- Includes all severities and both boundary-safe and internal-only codes

Intended for:

- Internal tooling
- Documentation generators
- Test/CI validation

### `error-registry.boundary-safe.json`

Subset of the registry that only includes codes with:

```json
{
  "meta": {
    "boundarySafe": true
  }
}
```

Recommended for:

- Public API / SDK error surfaces
- UI components that present errors across trust boundaries
- Remote logging and telemetry that should not leak internal-only error codes

### `error-registry.fatal-core.json`

Subset targeted at **core / native integrations**:

- Restricted to core domains (`env`, `backing`, `spec`, `plan`, `handoff`, `binding`)
- Restricted to errors with `severity: "fatal"` and `recoverable: false`

Good base for:

- C / C++ / Rust enums of “hard failure” conditions
- Host-level health checks and crash reporting

## How these files are generated

All exports are produced by `@seqlok/introspect` using:

- `src/errors/export-json.ts` – builds the in-memory `ErrorRegistryExport` object
- `scripts/export-error-registry.ts` – CLI wrapper that writes JSON to stdout or file

From **inside** `packages/introspect`:

```bash
# Full registry → schema/error-registry.full.json
pnpm run schema:errors:full

# Boundary-safe subset → schema/error-registry.boundary-safe.json
pnpm run schema:errors:boundary-safe

# Fatal-core subset → schema/error-registry.fatal-core.json
pnpm run schema:errors:fatal-core

# Regenerate all snapshots
pnpm run schema:errors:snapshots
```

From the **workspace root**, if wired in `package.json`:

```bash
pnpm schema:errors:snapshots
```

These commands will **overwrite** the snapshot files with fresh exports derived
from the current TypeScript registries.

---

## Intended consumers

- **TypeScript / JS** – should use the `@seqlok/introspect` APIs directly
  (`buildErrorRegistryExport`, `ALL_DOMAINS`, etc.).
- **Other languages** – should treat the JSON snapshot files as the canonical
  view of the error registry and derive native enums / types from them.
- **Docs / tooling** – should use `error-registry.boundary-safe.json` as the
  default source for anything user-facing, and opt into `full` only for
  internal or debug tooling.
